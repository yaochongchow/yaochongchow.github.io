Distributed Rate Limiter

A production-grade, fault-tolerant rate limiting service designed to scale horizontally without consistency trade-offs.

The Problem

Distributed APIs need rate limiting to prevent abuse and protect downstream services. A centralized limiter is a single point of failure. A purely local limiter can't enforce global request budgets across nodes.

Why It's Hard

Atomic state across distributed processes requires distributed locks or consensus — both expensive. The challenge: enforce consistent global limits with sub-millisecond overhead under 12K+ req/sec without creating a new bottleneck.

Key Trade-offs

  • Strong consistency via Lua — Redis executes token bucket logic atomically server-side, eliminating race conditions at the cost of Redis dependency
  • Stateless nodes — Go services hold zero local state; all rate limit state lives in Redis Cluster, enabling true horizontal scale
  • gRPC over HTTP — binary framing + multiplexing reduced per-request overhead vs REST at high concurrency

Optimizations

  • Lua scripting for atomic check-and-decrement in a single Redis round-trip
  • Redis pipelining to batch operations under burst load
  • Envoy health checks + circuit breaking for <50ms failover on node loss
  • Chaos testing: injected random node failures, measured no dropped legitimate requests

How I Think

The principles I bring to every system I build.

01

Reliability First

I design for failure before optimizing the happy path. Every component I build degrades gracefully — circuit breakers, retries, fallbacks are not afterthoughts.

02

Metrics-Driven Decisions

I don't optimize blind. Instrument first, identify the bottleneck with data, then optimize with measured impact. Intuition is a starting point, not a conclusion.

03

End-to-End Ownership

I trace problems from the client request to the database row. No black boxes. I own the full path — network, application logic, data layer.

04

Cross-Layer Debugging

Performance issues rarely live in one layer. I debug at the network, application, and data layer simultaneously — packet traces, query plans, flame graphs.

05

Performance as a Constraint

Latency and throughput aren't afterthoughts — they're design constraints I apply from day one, shaping architecture decisions before a line of code is written.
